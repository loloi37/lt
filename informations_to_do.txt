# 1. First: what “Security section” should actually contain

Your **Settings → Security** page should map to real backend logic.

### Minimum (you should have this 100%)

* **Change password**
* **Email change**
* **Sessions (devices logged in)**
* **Logout all devices**

### Strong (you should add this next)

* **2FA (two-factor authentication)**
* **Recent activity (logins, important actions)**
* **Account recovery (backup codes)**

### Advanced (fits your product specifically)

* **Trusted people (co-guardians / heirs)**
* **Access rules after inactivity (dead-man-switch logic)**

---

# 2. The real fix: centralize authorization (THIS is the big one)

Right now you repeat logic like:

* owner
* co_guardian
* witness
* reader

in many routes.

You need ONE system like:

```ts
function canUser(user, memorial, action) {
  // central logic
}
```

Example:

```ts
canUser(user, memorial, "edit_memorial")
canUser(user, memorial, "invite_member")
canUser(user, memorial, "export_archive")
```

### Why this matters

* You stop duplicating checks everywhere
* You avoid inconsistencies (huge security risk)
* You can evolve plans (family vs personal) easily

---

# 3. BIG rule: never trust client data (you must fix this everywhere)

Right now some routes accept:

```ts
userId
memorialId
```

That should NEVER come from the frontend.

Instead:

```ts
const { user } = await createAuthenticatedClient()
```

Then:

```ts
// derive everything from DB
```

Good:

```ts
user.id (from auth)
```

Bad:

```ts
userId from request body
```

# 4. Now your actual question:

## Should security differ between **Personal** and **Family plan**?

Yes — but not in the way you think.

---

## A. Personal plan → simple security

Think:

> “This is my private archive”

Security should be:

* Only owner can:

  * edit
  * export
  * delete
* No shared access
* No complex roles

Keep it **very strict + simple**

---

## B. Family plan → collaborative security

Think:

> “This is a shared memory space”

Now you need:

### Roles (you already have them)

* owner
* co_guardian
* witness
* reader

### But you must define CLEAR permissions:


Owner can edit the archive, invite users, add memories, approve content, and export the archive.

Co-guardian can edit the archive, invite users, add memories, and approve content, but cannot export the archive (this might be allowed in some cases).

Witness can only add memories and cannot edit the archive, invite users, approve content, or export the archive.

Reader cannot perform any actions (cannot edit, invite users, add memories, approve content, or export the archive).

This should exist in code (not just in your head)

## **Change the navigation bar about the plan** : 


## Personal plan

**Dashboard**
This should be the main control center for one memorial: progress, quick actions, status, and the main CTA to continue editing or preserve. Your current archive dashboard already uses action cards like “Read the archive,” “Share a memory,” and “Review contributions,” so for Personal it should feel like a simple hub for one archive.

**Memorial**
This should be the finished memorial view: the public-facing or readable archive page, with biography, photos, memories, and videos rendered from the saved memorial data. Your `render-data` route builds exactly that combined memorial dataset, and the view page renders it with `MemorialRenderer`.

**Edit**
This is the full memorial editor, where the owner changes the step-based content, saves updates, and creates history snapshots. The save route shows that owner or co-guardian edits update steps 1–9 and record version history.

**Preserve**
This should be the sealing / export flow: confirmation, authorization, payment, successor setup, and final archive generation. Your code already has a seal-confirmation page, a preservation/export route, and a succession component that can be required before permanent preservation.

**Settings**
This should hold account-level controls: profile, security, billing, and personal preferences. For your product, it should also include archive-level safety controls like plan status and maybe export/download preferences, because your app is tightly tied to ownership and preservation. That part is my recommendation rather than a route you already have.

---

## Family plan

**Overview**
This should be the family dashboard hub. It should show the archive summary, who has access, what is pending, and the main actions: read archive, add a memory, open family map, and open steward tools. Your current dashboard actions already point in that direction for family mode.

**Family members**
This should show the member list for the memorial: owner, co-guardians, witnesses, readers, and pending invitations. Your members route returns exactly that structure, and your invite/update-role routes show that Family plan is where role management really matters.

**Contributions**
This should be the review center for witness submissions and guardian moderation. It should show pending items, approved items, revision history, and actions like approve, reject, or request changes. Your role-data, review, and resubmit routes are already built around that workflow.

**Relations / Family map**
This should show the connected memorials and the family links between them. In your code, the family-map route only exists for family memorials and returns linked memorials with relationship labels and access status, so this is the right place for the visual family network.

**Succession**
This should manage what happens when the owner is gone: successor name, access level, dead-man switch timing, and transfer rules. Your succession component already supports read-only, editorial, and full-ownership access levels, and your FAQ says succession planning is part of the product.

**Settings**
For Family, this should still include account/security/billing, but also family-specific permissions and stewardship controls. In practice, this is where the owner would manage plan-level settings, access policy, and the safety rules around collaboration. That is the most logical product split from the code you already have.

---

### The simplest rule

Personal = **one archive, one owner, simple control**.
Family = **one archive network, many roles, collaboration and governance**.

3 : 


Make the invite flow much shorter.
The flow shown on your screen is too long: email → accept witness → login/register → confirm email → go back to the same invite link → then it works. That is too many steps and people will get lost. The user should land in one clear flow, not feel like they are fighting the system.


Fix notifications for memorial requests.
You already noticed the owner does not clearly see the request. That is a real product problem. There should be a visible notification center, a badge count, and a clear “Pending requests” section in the owner dashboard. Refreshing the page manually is not a solution.


Do not force biography text into a weird case rule.
If the biography is being forced to lowercase, I would not do that. It is bad for names, dates, acronyms, and readability. Better: keep the text natural, and only style the heading, not the content.


Media management needs a real structure.
Your note about photos/videos is correct. A memorial with many media items should have:
a gallery view
a grid preview
a detail drawer or modal when clicked
delete/edit actions only for the right roles


Right now this is one of the places that can become messy fast.
Add a confirmation step for destructive actions.
Delete photo, delete video, delete banner, delete childhood photo, remove profile image: all of those should require confirmation. And for the family plan, some of them should maybe only be allowed for owner/co-guardian.

Make mobile the default test, not an afterthought.
A memorial product like this will probably be used a lot on phones. I would check:
sidebar collapse
media grid spacing
long text wrapping
invite flow on small screens
buttons not too close together
Improve naming and consistency.


Add empty states and locked states.
For example:
no photos yet
no videos yet
no memorial request yet
no family members yet
Without these, the app feels broken when there is simply no data.


Add a real activity/history view.
This is a big one you may not be thinking about. For a product like this, users will want to know:
who changed what
who approved what
who requested what
when a memorial was updated
That becomes important for trust, especially in Family mode.
Separate personal vs family rules everywhere.
Not just in the navbar. The same idea should affect:
permissions
notifications
invite flows
content approval
preservation/export
Personal should feel simple. Family should feel collaborative and controlled.




The most important ones are:

Activity log / history: show who changed what, who approved what, and when. For a family product, this builds trust.
Undo / restore for deleted media: especially for photos and videos. Permanent delete is risky.
Clear draft vs published/preserved state: users should always know whether the memorial is still editable or already sealed.
Accessibility and mobile spacing: bigger tap targets, better contrast, simpler layout on small screens.
Empty states: when there are no photos, no members, no requests, or no contributions, the page should still look intentional.

So the answer is: yes, I would add more, but I would keep them focused on trust, safety, and clarity — not extra decoration.



Autosave + “saved / saving / unsaved changes” indicator. This is huge for long memorial forms. People should never wonder whether they lost text.

Search, filter, and sort for family members, contributions, and media. Once the archive grows, scrolling becomes painful.
A clear status badge everywhere like Draft, Awaiting approval, Preserved, Family, Personal. Users should understand the state at a glance.

A first-time onboarding checklist so new users know what to do next instead of landing on a confusing empty dashboard.

The strongest one from this list is autosave with visible save status. That will make the product feel much more reliable immediately.
